## What Is Delivered vs Foundation-Level

### Delivered (production-ready on main)

- Authenticated operator REST API under `/api/operator/`
- HTML dashboard at `/dashboard` with real-time metrics
- Archival CSV export plus authenticated CSV download with
  deployment-safe filename handling
- Settlement workflow persisted in SQLite with load-on-init recovery
- Settlement approve/deny/execute surfaces in API, dashboard, and Telegram
- Approved settlement execution can top up and auto-retry the original API call
- CI-enforced architecture invariants (26 blocking tests)
- Automated retention pruning (6h) + nightly data cleanup
- Telemetry auto-recording (hourly snapshots)
- Workflow + pipeline SQLite persistence
- Unified policy boundary (gateway, review, build callers migrated)
- Memory injection filtered by provenance + kind (truthfulness-first)
- Structured 400 responses on all operator API query params
- Telegram `/settlement`, `/settlement approve`, `/settlement deny`, and
  `/settlement execute` operator actions
- Delivery without approval queue is always denied, including review/build paths
- Project root no longer silently falls back to `~/.agent-life-space`
- Pidfile path is configurable through `AGENT_PIDFILE_PATH`
- Vault readiness is explicit through `is_ready`, and startup logs now surface
  config posture
- Docker availability is now runtime state, not an env-var side effect
- Gateway payment callback is wired at construction time instead of by
  post-init mutation
- Dashboard now requires API auth before serving operator HTML
- Settlement and archival now go through public control-plane methods instead of
  private storage access
- Reporting receives settlement dependency at construction time, not by
  post-init private mutation

### Still Not Fully Closed

- **Operator workflow depth**: dashboard and API are real, but there is still
  no richer interactive operator app, recurring workflow engine, or deeper
  drill-down UX.
- **Gateway breadth**: `obolos.tech` is now honest and usable, but multi-provider
  orchestration and broader downstream workflow/telemetry remain future scope.
- **Deployment contract**: hardening is materially better, but the whole stack
  is not yet enforced as one extraction-grade runtime contract.
- **Selective extraction**: module seams are much clearer, but future service
  extraction is still readiness work, not a completed program.

## Next Steps

1. make future service extraction obvious from module boundaries
2. reduce the remaining hidden coupling and implicit shared state
3. keep config, storage, and data-handling posture explicit as the runtime grows
